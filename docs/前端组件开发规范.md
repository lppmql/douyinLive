# 前端组件开发规范

## 适用范围

本规则适用于本项目所有前端页面的新建、修改、重构和视觉优化。

前端技术栈：

* Vue 3
* TypeScript
* Vite
* SoybeanAdmin
* Naive UI
* UnoCSS
* Pinia
* Elegant Router

SoybeanAdmin 完整示例分支：

`https://github.com/soybeanjs/soybean-admin/tree/example`

---

## 核心原则

开发前端功能时，不要立即从零编写组件。

必须遵循以下优先顺序：

```text
当前项目已有组件
→ SoybeanAdmin example 示例
→ Naive UI 原生组件
→ 封装项目业务组件
→ 最后才考虑新增第三方依赖
```

目标是：

* 减少重复代码；
* 保持页面风格统一；
* 复用成熟交互模式；
* 降低维护成本；
* 避免项目继续臃肿。

---

## 开发前必须检查

每次开发或修改前端页面前，先检查以下内容。

### 1. 检查当前项目

搜索：

```text
frontend/src/components
frontend/src/hooks
frontend/src/composables
frontend/src/views/*/modules
frontend/src/utils
frontend/src/store
```

确认项目中是否已经有：

* 相同组件；
* 相同表格逻辑；
* 相同分页逻辑；
* 相同状态标签；
* 相同加载、空数据、错误状态；
* 相同日期筛选；
* 相同主播信息展示；
* 相同任务进度展示；
* 相同接口数据转换。

已有实现能满足时，优先复用或扩展，不重复创建。

---

### 2. 检查 SoybeanAdmin example

在 SoybeanAdmin `example` 分支中寻找相似的：

* 页面布局；
* 数据表格；
* 搜索表单；
* 分页处理；
* 表格工具栏；
* 新增或编辑抽屉；
* 状态标签；
* 数据卡片；
* 响应式栅格；
* 空状态；
* 加载状态；
* 移动端适配；
* 表单校验；
* 页面拆分方式；
* Hook 或 composable。

重点参考的模式包括：

```text
useNaivePaginatedTable
useTableOperate
TableHeaderOperation
NCard
NGrid
NGi
NSpace
NDataTable
NDrawer
NModal
NForm
NTag
NAlert
NPopconfirm
CountTo
```

只参考适合本项目的结构和实现方式，不复制无关业务代码。

---

### 3. 检查 Naive UI

SoybeanAdmin 示例没有合适实现时，先检查 Naive UI 是否已经提供原生组件。

优先使用 Naive UI，避免为了一个简单交互重新引入组件库。

## 组件使用原则

### 通用功能直接复用成熟模式

以下需求应优先参考 SoybeanAdmin 示例：

| 需求      | 推荐模式                        |
| ------- | --------------------------- |
| 列表查询    | Search 组件 + Paginated Table |
| 分页表格    | `useNaivePaginatedTable`    |
| 表格列控制   | `TableHeaderOperation`      |
| 新增、编辑   | `OperateDrawer`             |
| 删除确认    | `NPopconfirm`               |
| 状态展示    | `NTag`                      |
| 日期和条件筛选 | 独立 Filter/Search Card       |
| 首页指标    | `NGrid + NCard + CountTo`   |
| 响应式布局   | `NGrid + NGi`               |
| 加载提示    | Skeleton 或 Spin             |
| 空数据     | Empty State                 |
| 接口异常    | Error State + 重试按钮          |

---

### 业务功能封装项目组件

SoybeanAdmin 中没有的直播业务组件，应在项目内封装。

例如：

```text
BusinessMetricCard
BusinessStatusTag
BusinessPageToolbar
BusinessFilterCard
BusinessEmptyState
BusinessErrorState
AnchorAvatar
TaskProgress
SessionSelector
DateRangeToolbar

CollectorAccountCard
CollectorTaskPanel
AsrRuntimeCard
LiveSessionMetricGrid
ReviewScoreCard
ReviewEvidenceCard
TranscriptTimeline
KnowledgeSourceCard
ScheduleWarningCard
```

业务组件外观应继续遵循 SoybeanAdmin 和 Naive UI 的布局、圆角、间距、颜色与响应式规范。

## 页面结构原则

页面 `index.vue` 只负责：

* 页面布局；
* 模块组合；
* 少量页面级状态；
* 调用 composable；
* 接收标准化后的 ViewModel。

不要在一个页面文件中同时放置：

* 大量接口请求；
* 多组轮询逻辑；
* 复杂数据转换；
* 业务规则判断；
* 大量表格列配置；
* 多个大型弹窗；
* 大段 CSS；
* 后端数据兼容逻辑。

推荐结构：

```text
views/example/
├── index.vue
├── modules/
│   ├── ExampleOverview.vue
│   ├── ExampleFilter.vue
│   ├── ExampleTable.vue
│   └── ExampleDrawer.vue
├── composables/
│   ├── useExampleData.ts
│   ├── useExampleTable.ts
│   └── useExamplePolling.ts
└── adapters/
    └── example-adapter.ts
```

职责：

```text
index.vue       页面布局和模块组合
modules         独立页面区域
composables     状态、请求、轮询和交互逻辑
adapters        接口数据转换为前端 ViewModel
components      跨页面复用组件
```

## 当前项目各页面参考方向

### 首页和经营仪表盘

* 响应式指标卡；

### 主播话术页面


### AI 复盘页面

建议结构：

```text

```

报告解析、字段兼容和 fallback 逻辑必须放在 adapter，不放在页面模板中。

---

### 知识库页面

保留聊天业务形态，优化：

* 推荐问题；
* 回答加载状态；
* 流式回答；
* 错误重试；
* 引用来源；
* 来源定位；
* 移动端来源抽屉。

不能为了套用通用聊天示例，破坏“每个回答可追溯到真实场次”的业务要求。

---

### 主播排班页面

参考搜索卡和表格页面：

* 日期筛选；
* 主播筛选；
* 状态概览；
* 排班表；
* 缺场、短时长、加场标签；

---

### 用户管理

优先参考 SoybeanAdmin `example` 分支用户管理示例：

* `useNaivePaginatedTable`

## 禁止事项

禁止为了“参考示例”而做以下事情：

* 整页复制 SoybeanAdmin 示例业务；
* 复制模拟接口；
* 引入无关图表库；
* 引入甘特图、地图、编辑器等无关依赖；
* 重复封装 Naive UI 已经提供的基础组件；
* 创建只使用一次且没有必要的公共组件；
* 为了拆分而机械拆分；
* 修改现有后端接口行为；
* 将生成的 OpenAPI 类型重新手写一份；
* 在多个页面重复编写相同状态和分页逻辑；
* 在页面组件内写复杂业务规则；
* 一次性重构全部页面。

---

## 新增公共组件的条件

新增公共组件前，应至少满足一项：

1. 已经有两个或以上页面需要使用；
2. 同类代码重复出现两次以上；
3. 组件包含统一视觉规范；
4. 组件能明显减少页面复杂度；
5. 组件是项目核心业务表达。

只被一个简单页面使用、且代码很少时，优先保留为页面内部模块。

---

## 依赖管理原则

新增第三方依赖前必须说明：

* 现有项目为什么不能实现；
* SoybeanAdmin 为什么没有可复用方案；
* Naive UI 为什么不能满足；
* 新依赖解决了什么问题；
* 包体积增加多少；
* 是否支持 Vue 3 和 TypeScript；
* 是否会影响构建和维护；
* 是否有更轻量替代方案。

没有充分理由时，禁止新增依赖。

---

## 接口和类型规则

* 后端已有的响应类型禁止重复手写；
* 优先使用 OpenAPI 自动生成类型；
* 后端 Schema 修改后必须执行：

```bash
cd frontend
pnpm gen-api
pnpm typecheck
```

* 页面不要直接处理不稳定的后端原始结构；
* 必要时通过 adapter 转换为稳定的前端 ViewModel；
* 数组字段必须合理处理空值，但不能用兜底掩盖后端契约错误。

---

## 每个页面完成后的检查

每完成一个页面，必须执行：

```bash
cd frontend
pnpm typecheck
pnpm exec oxlint .
pnpm exec eslint .
pnpm build
```

并人工检查：

```text
桌面端 1440px
笔记本 1280px
平板
手机
浅色模式
深色模式
正常数据
空数据
加载中

重复点击
```

## 开发输出要求

每次完成前端开发后，应输出：

1. 参考了 SoybeanAdmin example 中哪些组件或页面模式；
2. 复用了项目中的哪些现有组件；
3. 新增了哪些业务组件；
4. 删除了哪些重复实现；
5. 修改了哪些页面；
6. 是否新增依赖；
7. 类型检查结果；
8. Lint 结果；
9. 构建结果；
10. 移动端和深色模式验证情况。

---

## 最终开发准则

所有前端开发遵循：

```text
先复用
→ 再参考
→ 再封装
→ 最后才从零开发
```

参考顺序固定为：

```text
当前项目
→ SoybeanAdmin example
→ Naive UI
→ 项目业务组件
→ 第三方依赖
```

优化的目标不是让项目与 SoybeanAdmin 示例页面完全相同，而是：

> 使用 SoybeanAdmin 的工程规范和成熟交互模式，构建符合零食店直播运营业务的统一前端体验。
