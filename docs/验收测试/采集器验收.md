# 数据采集验收

## 前置条件

- 项目保存的采集账号已扫码登录，并通过“检查存活”。
- MySQL、Redis、后端和前端运行正常。
- 选择至少包含两个主播、两个真实 `roomId` 的账号进行验收。

## 操作步骤

1. 记录开始前主播数、场次数、评论数和最新场次时间。
2. 执行一次“刷新数据采集”，保留任务 ID 与 Trace ID。
3. 任务结束后再次执行相同刷新，验证相同 `roomId` 不产生重复场次。
4. 开启实时监控，确认真实开播主播被识别；停止监控后再启动，确认浏览器上下文可以恢复。
5. 随机抽查两个主播的最近场次，核对评论 `session_id`、平台时间和场次起止时间。

## 预期结果

- 主播、场次、评论和分钟指标只保存平台真实返回的数据。
- 相同平台场次重复采集保持幂等；不同主播或连续场次不会串场。
- 页面显示已发现、已同步、已补全和失败数量，失败日志包含 Trace ID 和可执行建议。
- 浏览器异常后复用保存的 Cookie、StorageState 和指纹恢复，不要求重复扫码。

## 失败与恢复

- 登录过期：重新扫码后先“检查存活”，再重试原任务。
- `Target page ... closed`：确认没有重复采集进程，保留 Trace ID 后重试。
- 评论串场：停止下游知识同步，先核对 `roomId + session_id + comment_time` 映射，再重新采集。
