# 领域文档

工程技能在探索代码库时应如何阅读本项目的领域文档。

## 探索前先读这些

- **`CONTEXT.md`**（项目根目录）——如果存在的话。
- **`docs/adr/`**——阅读与你即将处理的领域相关的架构决策记录（ADR）。

如果以上文件不存在，**直接跳过**。不要提示缺失，也不要建议提前创建。当术语或决策真正确定时，`/domain-modeling` 技能（通过 `/grill-with-docs` 和 `/improve-codebase-architecture` 触发）会按需创建它们。

## 文件结构

单上下文仓库（本项目）：

```

├── CONTEXT.md          ← 可能尚未创建
├── docs/adr/
│   ├── 0001-模块化单体架构与轻量可观测性.md
│   └── ...
└── src/
```

## 使用术语表中的词汇

当你的输出中提到某个领域概念时（Issue 标题、重构方案、假设、测试名称），使用 `CONTEXT.md` 中定义的术语。不要偏离到术语表明确避免的同义词。

如果你需要的概念不在术语表中，这是一个信号——要么你在发明项目不使用的语言（请重新考虑），要么确实存在空缺（记录下来供 `/domain-modeling` 使用）。

## 标记 ADR 冲突

如果你的输出与已有 ADR 矛盾，应明确表达而不是默默覆盖：

> _与 ADR-0007 矛盾（事件源订单）——但值得重新讨论，因为…_
