# ADR 0006: 项目维护标准与红线

**日期:** 2026-07-21  
**状态:** 已采纳  
**决策者:** 项目所有者

## 背景

项目功能持续增加，代码量增长，需要建立一套明确的维护标准来保证长期可维护性。当前存在以下问题：

- `manual_collect.py` 曾达 2,827 行（已拆分为 8 个模块，编排器 664 行），`analysis/index.vue` 达 1,000 行，`transcripts/index.vue` 达 795 行，仍需继续拆分
- 没有版本标签，无法快速回滚
- 缺少新开发者上手指南（开发.md、部署.md、故障排查.md）
- 部分 Service 同时操作数据库、Redis、浏览器和外部 AI

## 决策

采用以下维护标准和红线：

### 1. 开发流程固定化

每次改代码遵循统一流程：拉取 → 修改 → gen-api（如改 Schema）→ alembic（如改数据库）→ check → 提交 → 更新 CHANGELOG → 推送。

`make check` 一键执行：pytest + ruff + typecheck + lint + build + alembic check + docker compose config（CI 同样执行这些检查）。

### 2. 代码红线（软性标准）

| 指标 | 红线 | 说明 |
|------|------|------|
| Python 文件 | 500 行 | 超过评估拆分 |
| Vue 文件 | 400 行 | 超过评估拆分 |
| 单函数 | 80 行 | 优先拆分 |
| 函数参数 | 6 个 | 考虑用对象 |
| 类依赖 | 8 个组件 | 检查职责 |

不机械执行，每次代码评审检查。

### 3. 职责分层

```
api.py         → 只处理请求和响应
schemas.py     → 只定义接口数据结构
service.py     → 只处理业务逻辑
repository.py  → 只操作数据库（未来引入）
models.py      → 只定义 ORM 模型
```

一个 Service 不应同时操作：Playwright + MySQL + Redis + AI + ffmpeg。

### 4. 版本管理

使用语义版本号（v0.9.0），每次发布打标签。发布前备份数据库和关键文件，回滚方案提前明确。

### 5. 文档最小集

只维护 6 份核心文档：README、CHANGELOG、架构/README、开发、部署、故障排查。加上 ADR 记录架构决策。

## 后果

### 正面
- 新开发者能通过 开发.md 快速上手
- 发布和回滚有明确流程，减少操作失误
- 代码红线和职责分层指导重构方向
- `make check` 一次跑完所有检查，不会遗漏

### 负面
- 维护文档需要持续投入时间
- 代码红线可能导致"为了拆而拆"的过度工程
- 版本标签需要每次发布时手动维护

### 风险缓解
- 红线是"评估"而非"必须"，避免机械拆分
- 版本标签和 CHANGELOG 在提交时一并完成，不额外增加步骤
