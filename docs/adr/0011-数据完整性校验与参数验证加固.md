# ADR 0011：数据完整性校验与参数验证加固

- **日期**：2026-07-22
- **状态**：✅ 已执行

---

## 背景

经过两轮探索扫描，发现前后端在数据完整性保护上有 3 类缺口：

### 前端

1. **API 响应无运行时校验** — `backendRequest<T>()` 的泛型只在编译时有效，运行时如果后端返回了不一样的结构，前端会静默崩溃（白屏/数据不渲染）
2. **部分页面直接访问 `.data`** — `review-workbench.vue` 用 `response.data || null` 取值，不检查 `error` 字段，错误被吞掉
3. **请求参数无边界校验** — 分页参数传 0 或负数直接发给后端，导致数据库异常

### 后端

1. **大部分 CRUD schema 无字段约束** — `LiveRoomBase`、`CommentBase` 等 8 个 schema 的字符串字段没有 `min_length`/`max_length`，能提交 MB 级数据
2. **`LoginRequest` 接受空字符串** — `username=""` / `password=""` 能通过 Pydantic 校验
3. **查询参数缺下限** — `collector.py` 和 `ws.py` 的 `limit` 参数只有 `le=500` 没有 `ge=1`
4. **错误响应格式不统一** — 后端抛 HTTPException 返回 `{"detail":"..."}` ，但前端 `onBackendFail` 期望 `{"code":"XXXX","msg":"..."}` 格式

---

## 决策

### 前端 4 项

#### 1. 安全取值工具 `safeAccess.ts`

建三个零依赖的纯函数，像"安全气囊"一样工作：

- `safeGet(obj, path, defaultValue)` — 深层安全取值，任意一层为 null 则返回默认值
- `ensureArray(val)` — 确保返回值是数组，防止 `.map()` 崩溃
- `safeNumber(val, defaultVal=0)` — 安全数字转换，处理 null/字符串/NaN

#### 2. 响应拦截器加固

在 `backendRequest` 的 `transform` 函数里加 null/undefined 检查：
- 响应体为空 → 警告日志 + 返回 null（降级不白屏）
- `{code, data, msg}` 格式中 `data` 为空 → 警告日志 + 返回 null

#### 3. 复盘工作台修复

`review-workbench.vue` 中 5 处 API 调用从手动 `.data` 改为 `unwrapServiceData()`：
- `loadWorkbench` — 加载工作台数据
- `generateReview` — 生成复盘
- `updateFinding` — 更新发现状态
- `saveAsset` — 收录话术
- `changeAssetStatus` — 更新资产状态

修复前错误被静默吞掉（`response.data || null`），修复后错误会触发 toast 提示。

#### 4. API 参数边界校验

在 API 调用层对参数做范围裁剪：
- `fetchLiveSessionPage` — `current>=1`, `size 1~500`
- `fetchKnowledgeItemPage` — 同上
- `fetchKnowledgeTimeSlicePage` — 同上
- `syncRecentKnowledge` — `limit 1~500`
- `syncDataEase` — `limit 1~500`

### 后端 4 项

#### 5. CRUD Schema 字符串约束

为 8 个 Pydantic BaseModel 的 ~50 个字段加约束：

| Schema | 加的约束 |
|--------|---------|
| `LiveRoomBase` | 9 个字段加 `min_length=1, max_length=200~500` |
| `LiveSessionBase` | `room_id gt=0`，字符串加 `max_length=500~1000` |
| `CommentBase` | `session_id gt=0`，`is_high_intent ge=0,le=1`，字符串加约束 |
| `TranscriptSegmentBase` | `segment_start/end ge=0`，`text_content max_length=10000` |
| `LeadBase` | `is_valid ge=0,le=1`，字符串加约束 |
| `KnowledgeBaseBase` | `content max_length=50000`，字符串加约束 |
| `AnalysisReportBase` | `report_type min_length=1,max_length=100` |

#### 6. 登录请求约束

- `LoginRequest.username` — `min_length=1, max_length=100`
- `LoginRequest.password` — `min_length=1, max_length=128`
- `UserCreate.username` — `min_length=1, max_length=100`
- `UserCreate.password` — `min_length=6, max_length=128`
- `UserCreate.status` — `pattern="^(active|disabled)$"`
- `UserCreate.email/phone/nickname` — `max_length` 约束

#### 7. 查询参数范围

- `collector.py` — `limit: Query(100, ge=1, le=500)`
- `ws.py` — `limit: Query(200, ge=1, le=500)`

#### 8. 统一错误响应格式

新建 `error_handler.py`，注册 3 个 FastAPI 异常处理器：

| 异常类型 | 默认响应 | 统一后响应 |
|---------|---------|-----------|
| `HTTPException` | `{"detail":"..."}` | `{"code":"0404","msg":"...","data":null}` |
| `RequestValidationError` | `{"detail":[{...}]}` | `{"code":"0422","msg":"参数校验失败：...","data":null}` |
| `Exception` | HTML 500 页面 | `{"code":"0500","msg":"服务端内部错误...","data":null}` |

---

## 影响

- **前端更抗揍**：后端返回空数据/unexpected 结构也不会白屏，有日志可排查
- **后端拒绝垃圾数据**：空字符串/MB 级大文本在 Pydantic 层就被拦截，不会进入数据库
- **错误提示可达用户**：HTTPException 统一转 msg 字段，前端 toast 能正确显示
- **开发体验提升**：控制台有黄色警告标记异常数据，不用猜"为什么没数据"
