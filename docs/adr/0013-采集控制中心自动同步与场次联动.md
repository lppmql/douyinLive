# ADR 0013：采集控制中心自动同步与场次联动

- **日期**：2026-07-23
- **状态**：已执行
- **取代**：ADR 0012 的用户控制面与 ASR 资源策略

## 背景

ADR 0012 把刷新、监控、ASR、AI 复盘、知识库和 DataEase 都设计成用户开关。真实使用后发现，六个开关与共用进度条增加了理解成本，还允许用户误关必须持续同步的数据。与此同时，刷新和监控共用一个扫码账号、Cookie、浏览器指纹和 Playwright BrowserContext，重复启动后端或直接关闭上下文会中断正在运行的真实采集。

最终业务目标是：以“开零食店避坑知识科普、资料钩子和私信留资”为核心，保证直播数据持续进入系统，同时让本机资源占用可见、可控且可恢复。

## 决策

### 1. 用户只控制必须控制的动作

| 能力 | 最终入口 | 行为 |
|------|----------|------|
| 全部场次数据补齐刷新 | 手动按钮 | 高优先级执行，默认检查全部真实主播和全部场次 |
| 直播监控 | 长期开关 | 开启后持续发现开播并采集直播间数据，关闭后彻底停止对应循环 |
| ASR 话术转写 | 长期开关 | 优先处理正在开播的场次，否则从最新场次开始 |
| AI 复盘 | 场次详情按钮 | 由用户针对当前场次手动生成，避免无意消耗模型额度 |
| 知识库 | 后台自动 | 评论、话术、指标和复盘数据产生后自动增量写入 |
| DataEase | 后台自动 | 有完整场次数据后自动增量同步到专用只读宽表 |

采集页不再显示 AI、知识库和 DataEase 开关，也不再使用一个容易产生歧义的共用进度条。任务队列、资源概览和结构化日志分别回答“在做什么”“电脑是否吃得消”“具体发生了什么”。

### 2. 刷新优先接管共享浏览器

- 所有采集入口复用账号保存的 Cookie、StorageState 和浏览器指纹。
- 浏览器协调器只允许一个采集工作流持有租约，不同时创建相互争抢登录态的页面。
- 手动刷新优先级高于监控。刷新到来时，监控在安全检查点让出浏览器；刷新结束后监控自动恢复。
- 停止监控只停止监控循环，不关闭刷新正在使用的 BrowserContext。
- 后端退出先停止调度并等待安全检查点，不以强杀浏览器作为正常停止方式。

这里的“浏览器租约”是一个排队使用浏览器的凭证，用大白话说，就是刷新和监控轮流使用同一个已登录浏览器，不能同时抢方向盘。

### 3. ASR 使用实时资源计划

- 资源服务持续采样 CPU、内存和 CPU 核心数。
- 资源正常时，并发数可在配置的安全上限内自动提高，不再固定单任务。
- 资源偏高时降为单任务；内存达到严重阈值时不接新分片，资源恢复后继续。
- 调度顺序为“正在开播优先、最新场次优先”，已完成分片可断点续传。
- 关闭 ASR 开关后停止运行时和新任务创建，释放 FunASR、FFmpeg 等本机资源。

### 4. 后处理默认完整且幂等

- 知识库和 DataEase 每轮默认处理全部待同步场次，批次值 `0` 表示不设置 20 场上限。
- 同一个场次重复同步时按稳定业务键更新，不重复制造知识切片或宽表记录。
- AI 复盘不属于自动后处理，只在场次详情页由用户明确触发。
- Prompt 统一放在 `backend/app/prompts/`，业务服务通过注册表读取，便于版本追踪和测试。

“幂等”表示同一批真实数据重复执行不会越存越多，而是得到相同结果。

### 5. 页面围绕同一直播场次协作

- 直播场次详情、主播话术、AI 复盘和知识库通过 `sessionId` 保持同一场次上下文。
- 共享主播身份组件统一展示头像、昵称和抖音号；缺失字段使用明确兜底，不留空也不猜测。
- 评论严格按 `session_id` 查询并展示真实用户昵称。
- 企业评论接口没有返回头像或公开抖音号时显示“未获取”，不得把内部 `secUId` 冒充抖音号。

### 6. 任务和启动过程可恢复

- 持久任务保留停止、重试、心跳、Trace ID 和结构化结果。
- 调度器启动后回收由不同旧进程遗留且心跳超时的监控任务，避免假“运行中”。
- `start.sh` 使用原子目录锁阻止第二个启动编排器重复杀端口。
- 后端与前端由 `start.sh` 放入独立进程组，终端信号先由启动脚本接收，再按安全顺序通知业务进程。
- 正常重启给予后端和前端最多 60 秒优雅退出时间；只有确实卡死才执行最终强制清理。
- `.runtime/` 只保存本机运行锁并由 Git 忽略，不包含业务数据。

### 7. 原生媒体使用短时只读会话

- 普通业务 API 继续只接受 Bearer Token，不能因头像或视频加载困难而整体取消鉴权。
- 登录、刷新 Token 和获取用户信息时，后端额外签发短时 HttpOnly 媒体 Cookie。
- 媒体 Cookie 只允许读取主播头像、评论头像、回放和视频下载，不能调用列表、采集、复盘等业务 API。
- Cookie 不可被前端 JavaScript 读取，主登录 Token 不拼接到媒体 URL，避免被代理日志、浏览器历史或 Referer 记录。
- 页面响应解析任务在关闭 Playwright 页面前统一回收，避免浏览器驱动退出后出现未读取的 Future 异常。

## 真实数据边界

2026-07-23 使用项目保存的真实账号检查企业评论接口。接口返回评论内容、时间、昵称、`secUId` 和平台内部用户 ID，但没有头像 URL 或公开抖音号。因此当前系统只能保证评论昵称真实；头像和公开抖音号必须等上游接口真实提供后再保存，不能伪造或推断。

同日企业账号来源返回 10 个唯一主播资料，而不是预期的 9 个。其中包含一个只有极短历史场次的账号。系统保留上游返回的 10 个真实身份，不在没有业务确认的情况下删除或合并。

## 影响

### 正面

- 用户操作减少，知识库和 DataEase 不会因误关而漏同步。
- 手动刷新与实时监控共享登录态但不再争抢浏览器。
- ASR 能利用空闲资源，也能在电脑压力升高时自动退让。
- 所有分析页面围绕当前场次联动，减少选错主播或场次。
- 重复启动和旧任务残留不再轻易破坏正在进行的采集。

### 代价

- 浏览器采集仍按安全顺序执行，刷新期间监控会短暂等待。
- 自动知识库与 DataEase 同步不可在页面关闭，只能通过运维配置停用后台调度。
- 上游接口缺少评论头像和公开抖音号时，前端只能诚实显示未获取。

## 验证

2026-07-23 使用真实 Cookie、浏览器指纹、MySQL 和企业平台接口完成验收：

- Cookie 存活检查 HTTP 200，真实账号昵称与抖音号可读取。
- 补齐刷新任务 `#7443` 只执行一次，读取企业账号记录 10 个，发现 1032 场，新增 1 场，补齐 1 场，失败 0 场。
- 任务阶段完整经过浏览器接管、准备、登录检查、房间采集、企业同步、历史同步、详情补齐、Cookie 刷新和完成。
- 稳定刷新过程中没有浏览器关闭、上下文关闭或页面句柄冲突日志，刷新完成后监控继续运行。
- 实时监控识别场次 `#13370` 正在开播；在该场采集任务运行时执行停机，系统等待 12 条趋势、9 条新增评论和 36 条画像成功写入后再释放浏览器，没有遗留端口或异步 Future 错误。
- 后端 233 项测试通过，覆盖率 52.24%；数据库迁移、Docker 配置、前端 lint、类型检查与生产构建全部通过。
- 前端真实 Playwright 冒烟测试 10 项通过，覆盖采集页和“详情→话术→复盘→知识库”场次联动。

## 替代方案

| 方案 | 优点 | 未采用原因 |
|------|------|------------|
| 保留六个开关 | 用户能控制每个模块 | 容易误关强制数据链路，页面理解成本高 |
| 刷新和监控各用一个浏览器 | 表面并行更快 | 同账号并发登录风险高，Cookie 和指纹易冲突 |
| ASR 永久固定单并发 | 资源最保守 | 资源充足时处理过慢，无法满足最新场次优先 |
| ASR 无限制并发 | 转写速度快 | FFmpeg 和模型可能迅速耗尽内存导致电脑卡死 |
| 自动生成所有 AI 复盘 | 无需人工操作 | 模型成本不可控，低价值场次也会消耗额度 |
