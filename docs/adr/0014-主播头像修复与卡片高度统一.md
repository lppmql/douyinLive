# ADR 0014 — 主播头像修复与卡片高度统一

**日期**：2026-07-23
**状态**：已采纳
**决策者**：lppmql

---

## 背景

用户反馈两个前端展示问题：

1. **主播头像不显示**：首页主播明细表中部分主播头像为空，即使该主播在直播场次中有过真实头像 URL。
2. **控制中心卡片高度不齐**：数据处理控制中心 6 个模块卡片和 4 个电脑资源统计卡在同一行高度不一致，视觉观感差。

---

## 方案对比

### 问题 1：主播头像不显示

#### 根本原因

后端 `/dashboard/summary/by-anchor` 接口使用了 `GROUP BY` 多个字段：

```sql
GROUP BY douyin_id, anchor_name, anchor_avatar_url
```

同一个主播在多场直播中 `anchor_avatar_url` 可能不同（有的场次有头像，有的没有），导致该主播被拆成多行数据。前端只显示第一条数据，那一条的 `anchor_avatar_url` 可能恰好为空。

#### 方案 A：子查询 + JOIN（✅ 已选择）

- 子查询只按 `douyin_id` GROUP BY，聚合统计数据，不碰 `anchor_avatar_url`
- 主查询 JOIN 最新场次（`MAX(id)`）获取主播名和头像 URL
- **优点**：SQL 语义正确，数据唯一，头像取最新场次的值
- **缺点**：多一次 JOIN 开销（主播数量级很小，可忽略）

#### 方案 B：子查询取最新非空头像

- 子查询用 `MAX(anchor_avatar_url)` 或子查询取最新非空值
- **优点**：简单
- **缺点**：`MAX(anchor_avatar_url)` 可能取到旧头像；子查询取最新非空逻辑复杂

**选择方案 A**，因为它保证了数据语义正确且一行一主播。

### 问题 2：前端头像代理逻辑

原逻辑：

```typescript
const displayAvatarUrl = computed(() => {
  if (!props.avatarUrl) return '';  // ← 没有 avatarUrl 就不走代理
  return props.sessionId ? proxy : props.avatarUrl;
});
```

**问题**：如果 `avatarUrl` 为空，即使有 `sessionId` 也不会走后端代理（后端代理可以从数据库查到最新头像 URL）。

**修复后**：

```typescript
const displayAvatarUrl = computed(() => {
  if (props.sessionId) return getLiveSessionAvatarUrl(props.sessionId);  // 优先走代理
  if (props.avatarUrl) return props.avatarUrl;
  return '';
});
```

`sessionId` 存在时始终走后端代理，即使前端传的 `avatarUrl` 为空，后端也能从最新场次记录中查到。

### 问题 3：卡片高度统一

#### 方案 A：固定高度（❌ 不选）

- 给每个卡片设固定 `height` 值
- **缺点**：内容长度不确定时要么溢出要么留大片空白，响应式下问题更多

#### 方案 B：CSS Grid stretch + flex 列（✅ 已选择）

- 每行卡片加 `height: 100%` + `display: flex; flex-direction: column`
- 依赖 CSS Grid 默认 `align-items: stretch`，同一行所有卡片自动撑到最高卡片的高度
- **优点**：自适应，不依赖固定值，内容少的卡片也能填满整行高度
- **缺点**：无

**选择方案 B**，用 CSS Grid 原生能力实现，无需 JS 计算。
