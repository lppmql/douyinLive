# ADR 0012：数据采集六模块控制中心与持久任务队列（已被取代）

- **日期**：2026-07-22
- **状态**：已被 ADR 0013 取代

> 本文记录 2026-07-22 的初版方案，供追溯设计演进使用，不代表当前产品行为。当前实现以 [ADR 0013](./0013-采集控制中心自动同步与场次联动.md) 为准。

---

## 背景

原数据采集页把历史刷新、实时监控、ASR、AI 复盘、知识库和 DataEase 的状态分散在多个卡片中，部分后处理还会被采集任务隐式触发，存在以下问题：

1. 用户无法判断每个能力能否单独开启或停止。
2. 多个重任务同时运行时会争抢数据库、浏览器和本机内存。
3. 页面关闭或后端重启后，任务状态和进度不容易恢复。
4. 账号列表缺少真实扫码昵称、Cookie 检查时间和刷新时间。
5. 采集日志缺少主播、场次和阶段上下文，详情只能查看原始 JSON。
6. 直接关闭浏览器上下文容易产生 `Target page, context or browser has been closed` 错误。

## 决策

### 1. 六个模块分别控制

数据采集页固定提供六个独立开关：

| 模块 | 类型 | 开关行为 |
|------|------|----------|
| 全部数据刷新 | 一次性任务 | 执行全部主播、场次、评论、趋势和画像刷新 |
| 直播监控 | 常驻服务 | 定时发现开播并持续采集实时数据 |
| ASR 话术转写 | 常驻服务 | 单并发处理最新回放，支持断点续传 |
| AI 复盘 | 一次性任务 | 只处理待复盘的真实场次 |
| 存知识库 | 一次性任务 | 增量写入评论、话术、复盘和指标知识 |
| DataEase 同步 | 一次性任务 | 增量同步完整场次到只读宽表 |

全部数据刷新不再隐式启动 ASR、AI、知识库或 DataEase。用户可以按电脑资源和业务需要分别决定何时执行。

### 2. 重任务进入同一个持久队列

- 全部数据刷新、AI 复盘、知识库和 DataEase 先写入 MySQL，再由一个后台循环按顺序执行。
- 任务支持停止、失败重试、服务重启恢复、心跳、Trace ID 和结构化结果。
- 停止操作只设置取消标记，执行器在批次或浏览器操作的安全检查点退出，不强杀进程。
- 页面只显示一个共用进度条，当前任务结束后自动显示下一项任务。

这里的“队列”是按顺序等待执行的任务清单，用于防止多个重任务同时占满电脑资源。

### 3. 监控和刷新不再互相关闭

- 直播监控和全部数据刷新可以同时保持开启状态。
- 两者需要操作 Playwright 时，使用同一个异步锁排队复用浏览器登录态。
- 刷新占用浏览器期间，监控等待安全窗口；刷新结束后监控继续下一轮。
- 停止监控不再关闭正在执行刷新任务使用的浏览器上下文。

### 4. Cookie 与浏览器指纹作为账号资产保存

- 扫码成功后保存 StorageState、Cookie 备份、真实抖音昵称、抖音号和浏览器指纹。
- 状态文件权限固定为 `0600`，只有当前系统用户可以读写。
- “检查 Cookie”使用账号原指纹做真实页面访问；验证成功后保存平台轮换的新 Cookie。
- 账号列表展示扫码时间、Cookie 刷新时间和最近检查时间。

### 5. 日志结构化并默认脱敏

- 日志保存任务、阶段、事件类型、主播、场次、房间和数据详情。
- Cookie、Authorization、Token、直播流地址等敏感字段进入日志前统一移除。
- 前端把嵌套数据转换成中文键值列表，不向业务用户展示代码格式 JSON。
- 日志文件和数据库日志都设置数量或大小上限，避免长期采集占满磁盘。

### 6. ASR 使用离线低资源模式

- FunASR 使用离线模式，容器限制 2 核和 1.8 GB 内存。
- Worker 固定单并发，队列设置容量上限。
- 关闭 ASR 时把处理中任务恢复为排队状态，保留已完成音频分片，重新开启后断点续传。

## 影响

### 正面

- 业务人员能明确控制每一步处理，不再被隐式任务消耗资源。
- 重任务不会同时争抢电脑，页面关闭后任务仍然存在。
- 停止和重试有明确反馈，已经写入的真实数据不会丢失。
- 账号与日志信息更完整，问题可以定位到具体主播和场次。
- DataEase 继续使用专用只读账号，不扩大数据库权限。

### 代价

- 任务表、账号表和日志表增加了少量状态字段。
- 一次性重任务需要排队，后创建的任务可能需要等待。
- 旧日志没有完整上下文，只能保证新日志按新格式写入。

## 验证

初版验证结果不再作为当前验收基线。2026-07-23 已按最终产品逻辑重新使用真实 Cookie 和浏览器指纹验收，结果记录在 ADR 0013 与 `CHANGELOG.md`。

## 替代方案

| 方案 | 优点 | 未采用原因 |
|------|------|------------|
| 六个模块完全并行 | 完成速度可能更快 | 容易同时占用浏览器、CPU、内存和数据库连接 |
| 每次任务使用独立浏览器 | 隔离更彻底 | 同账号并发登录风险更高，内存占用明显增加 |
| 继续由刷新任务自动触发全部后处理 | 操作按钮更少 | 用户无法控制资源和成本，失败边界不清晰 |
| 引入 Celery 等外部任务平台 | 分布式能力强 | 当前单机项目复杂度和运维成本过高 |
